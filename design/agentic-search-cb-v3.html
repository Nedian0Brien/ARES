이번 비교에서는 KID를 1차 지표로, CLIP-sim을 보조로 사용합니다. Personalization 시나리오 기준선은 5-shot으로 잡겠습니다 — DreamBooth3, Custom Diffusion4, OFT-S5 모두가 공통적으로 보고한 설정입니다.

왜 Chat 뷰인가
에이전트는 한 질문에 대해 여러 분 동안 여러 단계를 거칩니다. 결과 카드 스트림으로는 "지금 어디까지 왔는지"를 직관적으로 알기 어렵습니다.
  • 시간 흐름이 자연스럽게 표현됨
  • 사용자가 언제든 끼어들 수 있는 채널이 항상 열려있음
  • 긴 분석 텍스트를 prose로 자연스럽게 흐르게 함
버블이 없는 이유
버블은 짧고 감정적인 메시지에 최적화된 패턴입니다. 연구 분석 텍스트는 길고 구조적이므로 버블에 가두지 않습니다. 흰 배경 위 prose가 research workbench 톤을 유지합니다.
구분 원칙: 사용자 입력은 다크 버블 (consumer chat 메타포 유지) / 에이전트 응답은 prose (도구 메타포).
사용자 개입 모델
에이전트가 작업 중이어도 입력창은 항상 활성. 메시지를 보내면 현재 phase 컨텍스트 + 선택된 카드가 함께 전달돼 에이전트가 계획을 즉시 갱신합니다.
  • Pause: 현재 도구 호출 완료 후 정지
  • 카드 클릭: 다음 turn의 컨텍스트로 추가
  • Shift ↵: 줄바꿈, ↵: 보내기
Phase 구분
4-에이전트 architecture (Reader · Reproduction · Experiment · Analyst) 에 맞춰 phase divider로 분할. 각 phase는 자체 도구 화이트리스트를 가지며, divider tag 색이 ARES 시스템 토큰을 따른다.
Action line
에이전트의 모든 tool_use를 1줄짜리 칩으로 노출. 사용자가 "무슨 검색을 했는지" 즉각 파악.
  • 구조: [icon] tool_name "args" [count] [status]
  • status: ✓ ok / run(pulse) / err
  • 실패 시 클릭 → 재시도 다이얼로그
Snippet card
검색 결과 후보. 클릭 = "다음 질문에 컨텍스트로 포함". 선택 상태는 녹색 보더 + ✓.
의도: 사용자가 에이전트의 컨텍스트 윈도우를 직접 편집하는 경험. ctx-bar에 chip으로 누적되어 항상 가시화.
Claim card / Conflict card
에이전트가 중요한 발견·충돌을 명시적으로 보고하는 체크포인트.
  • Claim (녹색): 합성된 핵심 주장 — 신뢰도와 출처를 함께 표시
  • Conflict (앰버): 소스 간 측정·정의 충돌 — 다음 단계에서 어떻게 해소할지 명시
두 카드는 사용자가 "이 발견이 맞는지" 직접 검증할 수 있게 돕는 신뢰 장치입니다.
Live streaming
에이전트가 token streaming 중이면 마지막 <p> 끝에 깜빡이는 cursor (▍). working-line은 spinner + 현재 작업 요약.
메시지 모델
Message는 role(user/agent/system) + blocks[]로 구성. block 종류:
  • prose — 마크다운 텍스트
  • action — tool_use 단일
  • snippet_row — Source 후보 N개
  • claim, conflict — 합성 카드
  • working — 현재 진행 상태
스트리밍 프로토콜
SSE / WebSocket: { type, msgId, blockId, delta } 이벤트 시퀀스. 클라이언트는 blockId 단위로 lookup 후 textNode append. tool_use의 status는 start → complete 또는 error로 갱신.
컨텍스트 선택
카드 선택 = selectedSourceIds: Set<string> 상태. 다음 사용자 메시지 전송 시 payload에 포함:
  • selectedSources: [{id, title, snippet}]
  • currentPhase, runId
에이전트는 이를 system prompt 프리픽스에 주입하여 다음 plan을 갱신합니다.
Phase 라이프사이클
phase 전환 시 클라이언트는 새 phase_divider block을 받음. phase_completed 이벤트는 시간 stats(1m 04s) 포함.
개요
이 그래프는 에이전트의 탐색 체인을 인과 그래프로 시각화합니다. 채팅이 시간 스트림이라면, 그래프는 공간 구조를 봅니다. 좌→우 방향이 인과 흐름.

  • Thinking — 에이전트가 <thinking> 블록을 방출. 하나 이상의 Search로 분기 가능.

  • Answer — 에이전트가 sub-question 합성을 finalize. supporting Source들이 녹색 점선으로 연결.
엣지 의미

  • in-progress — 파란 점선. 실행 중일 때만, 완료 시 leads-to로 전환.
  • found — Search→Source. 회색 점선.
