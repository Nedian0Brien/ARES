# ARES 구현 기획서 / 기능 명세서

Agentic Research Experimentation System

프로토타입 기준 정렬 버전

## 1. 문서 목적

이 문서는 `design/ARES Prototype.html`에 구현된 현재 프로토타입을 기준으로 ARES의 기능 명세를 정리한 문서다.

기존 기획 문서는 비전, 문제 정의, 확장 로드맵 중심이었지만, 현재 프로토타입은 이미 다음 항목을 구체적으로 표현하고 있다.

- 제품의 핵심 워크플로우 단계
- 각 화면의 정보 구조
- 에이전트가 개입하는 지점
- 주요 액션과 상태 표현 방식
- 워크스페이스 셸, 탐색 구조, 단축키

따라서 이 문서에서는 프로토타입을 우선적인 source of truth로 간주한다. 기존 구상과 프로토타입이 충돌하는 경우에는 프로토타입 구현을 우선한다.

## 2. 제품 정의

ARES는 논문 탐색, 리딩, 재현 연구, 실험 비교, 인사이트 정리, 글쓰기까지를 하나의 연속된 연구 워크플로우로 연결하는 에이전트 기반 연구 실행 워크스페이스다.

제품이 지향하는 기본 흐름은 다음과 같다.

`Search -> Reading -> Research -> Result -> Insight -> Writing`

이 정의는 기존 문서의 추상적 흐름인 `Paper -> Understanding -> Reproduction -> Experimentation -> Insight -> Next Research`를 실제 UI와 인터랙션 단위로 구체화한 것이다.

## 3. 제품 범위

현재 프로토타입 기준 MVP 범위는 "논문 하나를 선택한 뒤, 읽고, 재현 가능성을 점검하고, 실험 결과를 비교하고, 인사이트와 후속 가설을 정리해 글쓰기 초안으로 연결하는 것"이다.

즉, ARES의 초기 제품은 단순 논문 검색기나 요약기가 아니라 다음을 연결하는 연구 실행 도구다.

- 논문 후보 탐색
- 구조화된 논문 리딩
- 재현 준비와 체크리스트 관리
- 실험안 생성과 결과 비교
- 인사이트 및 후속 가설 축적
- 연구 문서 초안 작성

## 4. 핵심 사용자

### 4.1 대학원생 및 개인 연구자

관심 주제의 논문을 빠르게 추려 읽고, 실제 재현과 변형 실험으로 이어가려는 사용자

### 4.2 AI/ML 연구 엔지니어

논문을 검토한 뒤 내부 실험과 결과 비교까지 짧은 루프로 수행해야 하는 사용자

### 4.3 소규모 연구팀

프로젝트 단위로 논문, 실험, 인사이트를 누적하고 공유하려는 팀

현재 프로토타입에는 대규모 협업 기능은 없지만, 프로젝트와 워크스페이스 단위 구조는 이미 반영되어 있다.

## 5. 정보 구조

ARES는 워크스페이스 셸과 6단계 워크플로우 화면으로 구성된다.

### 5.1 워크스페이스 셸

모든 화면에서 공통으로 유지되는 상위 레이아웃은 다음 요소를 포함한다.

- 워크스페이스 전환 영역
- 글로벌 검색 진입점
- 새 논문 생성 진입점
- 프로젝트 목록
- 워크플로우 단계 네비게이션
- 상단 컨텍스트 바
- 모바일 하단 네비게이션

### 5.2 워크플로우 단계

- `Search`: 논문 탐색 및 수집
- `Reading`: 논문 구조화 리딩
- `Research`: 재현 준비 및 실험 설계
- `Result`: 실험 결과 비교
- `Insight`: 인사이트와 후속 가설 정리
- `Writing`: 연구 문서 초안 작성

## 6. 공통 UX 규칙

### 6.1 내비게이션

- 데스크톱에서는 좌측 사이드바를 통해 단계 이동
- 모바일에서는 하단 탭 네비게이션으로 단계 이동
- `Cmd/Ctrl + 1~6` 단축키로 각 단계 직접 이동

### 6.2 상태 표현

시스템은 작업 상태를 명시적으로 드러낸다.

- `done`
- `running`
- `todo`
- `queue`

이 상태 체계는 리딩 진행도, 재현 체크리스트, 실험 큐, 에이전트 작업 상태 전반에 공통적으로 사용된다.

### 6.3 에이전트 노출 방식

에이전트는 숨겨진 백엔드 기능이 아니라 UI 안에서 현재 단계의 책임 주체로 드러난다.

- Search: `Scout agent`
- Reading: `Reader agent`
- Research: `Reproduction agent`, `Experiment agent`, `Analyst agent`
- Result: `Analyst report`
- Insight/Writing: 가설 생성 및 초안 생성 액션

## 7. 화면별 기능 명세

### 7.1 Search

Search 화면은 연구 주제를 입력하고 관련 논문을 탐색하는 시작점이다.

#### 목적

- 연구 주제에 맞는 논문 후보 수집
- 결과를 빠르게 필터링 및 우선순위화
- 개별 논문을 저장하거나 Reading 단계로 전달

#### 주요 UI 구성

- 좌측 필터 패널
- 중앙 결과 리스트
- 우측 논문 미리보기 패널

#### 주요 기능

- 검색어 입력 기반 논문 탐색
- `Venue`, `Year`, `Relevance` 기준 필터링
- 라이브러리 상태 표시
  - 저장된 논문 수
  - 신규 논문 수
- 결과 리스트에서 다음 메타데이터 표시
  - 제목
  - 저자
  - 학회/출처
  - 코드 유무
  - 저장 여부
  - relevance 점수
- 결과 항목 선택 시 우측 preview 갱신
- preview에서 다음 정보 표시
  - abstract
  - key points
  - 연결된 코드 저장소
- 주요 액션
  - `Save`
  - `Read`

#### 에이전트 역할

Scout agent가 논문 검색 결과를 수집하고 정렬한 상태로 표현된다.

### 7.2 Reading

Reading 화면은 논문 본문을 구조적으로 읽고 재현에 필요한 정보를 추출하는 단계다.

#### 목적

- 논문을 섹션 단위로 읽기
- 핵심 주장, 결과, 한계를 하이라이트
- 재현에 필요한 파라미터를 정리

#### 주요 UI 구성

- 좌측 리딩 진행 패널
- 중앙 논문 리더
- 우측 Reader agent 패널

#### 주요 기능

- 섹션별 읽기 진행도 표시
  - Abstract
  - Introduction
  - Method
  - Experiments
  - Analysis
  - Conclusion
- 하이라이트 타입 분류
  - Method
  - Result
  - Limit
  - Claim
- 논문 본문 내 강조 표시
- Reader agent 요약 카드 제공
- 재현에 필요한 주요 파라미터 정리
  - Dataset
  - Model
  - Threshold 등 핵심 값
  - GPU 요구사항
- `Send to Research` 액션으로 다음 단계 연결

#### 에이전트 역할

Reader agent는 논문 구조화 요약, 재현 핵심 정보 추출, 중요 문장 하이라이트를 담당한다.

### 7.3 Research

Research 화면은 재현 준비와 실험 설계를 함께 다루는 실행 단계다.

#### 목적

- 논문의 재현 가능성을 점검
- 환경 설정과 체크리스트를 관리
- 기본 재현 실험과 추가 실험안을 운영

#### 주요 UI 구성

- 좌측 재현 패널
- 중앙 실험 관리 패널

#### 주요 기능

- 재현 가능성 점수 표시
- 연결된 코드 저장소 표시
- 재현 체크리스트 제공
  - 공개 코드 저장소 확인
  - 환경 의존성 설치
  - CUDA 버전 확인
  - 데이터셋 전처리
  - 체크포인트 로드
  - baseline 실행
  - 논문 수치와 결과 비교
- setup command snippet 제공
- 실험 목록 테이블 제공
  - 실험명
  - metric
  - status
  - result
- `New experiment`로 추가 실험 생성
- ablation suggestion 제안
- baseline 결과와 논문 결과의 delta를 간단 분석

#### 에이전트 역할

- Reproduction agent: 코드 저장소, 환경, 체크리스트 관리
- Experiment agent: 실험안 생성과 실행 큐 관리
- Analyst agent: baseline 차이 해석

### 7.4 Result

Result 화면은 논문 수치와 재현 결과를 비교해 차이를 해석하는 단계다.

#### 목적

- 핵심 지표를 한눈에 비교
- 실험별 결과 테이블을 정리
- 차이의 원인을 요약

#### 주요 UI 구성

- KPI 카드
- 결과 비교 테이블
- Analyst report 패널

#### 주요 기능

- 요약 KPI 표시
  - Paper
  - Reproduction
  - Delta
- 실험 결과 테이블 비교
  - Experiment
  - Metric
  - Paper
  - Ours
  - Delta
- 결과 상태 태그 표시
- Analyst 리포트에서 차이 원인 정리
- 핵심 인사이트 태그화

#### 에이전트 역할

Analyst agent가 결과 차이, 민감도, 전처리 불일치와 같은 해석을 제공한다.

### 7.5 Insight

Insight 화면은 결과 해석을 구조화된 연구 자산으로 전환하는 단계다.

#### 목적

- 재현 및 실험 결과에서 핵심 인사이트를 추출
- 후속 실험 및 연구 가설을 생성
- 사용자 메모를 연구 자산으로 남김

#### 주요 UI 구성

- 핵심 인사이트 카드 영역
- 후속 연구 가설 영역
- 우측 연구 메모 패널

#### 주요 기능

- 검증된 인사이트 목록 표시
- 후속 연구 가설 목록 표시
- `연구 제안서 초안 생성` CTA 제공
- 연구 메모 편집
- 메모 기반 액션
  - `실패 자산화`
  - `후속 가설`
  - `내보내기`

#### 에이전트 역할

이 단계에서는 Analyst의 해석이 연구 인사이트로 승격되며, Proposal 성격의 기능이 후속 가설과 제안서 초안 생성 액션으로 이어진다.

### 7.6 Writing

Writing 화면은 축적된 결과와 인사이트를 실제 연구 문서 초안으로 연결하는 단계다.

#### 목적

- 논문/리포트 섹션 단위 초안 작성
- AI 제안을 받아 문장을 개선
- 결과물을 외부로 내보낼 준비를 함

#### 주요 UI 구성

- 좌측 섹션 네비게이터
- 중앙 문서 편집/미리보기 영역
- 상단 액션 바

#### 주요 기능

- 문서 섹션 탐색
  - Abstract
  - Introduction
  - Related Work
  - Method
  - Experiments
  - Conclusion
- 섹션별 상태나 분량 표시
- 문서 통계 표시
  - Words
  - Pages
  - Refs
- `AI draft` 액션
- `Export` 액션
- 인라인 AI 제안 및 `Accept` 액션

#### 에이전트 역할

Writing 단계는 Proposal/Writer 성격의 보조 기능이 작동하는 영역이며, 현재 프로토타입에서는 명시적 agent label보다 AI-assisted drafting 경험으로 표현된다.

## 8. 에이전트 구조 명세

프로토타입에 정렬된 현재 에이전트 구조는 다음과 같다.

### 8.1 Scout Agent

- 논문 수집
- 검색 결과 큐 생성
- 우선순위 정렬

### 8.2 Reader Agent

- 논문 섹션별 구조화 요약
- 핵심 주장, 결과, 한계 하이라이트
- 재현 파라미터 추출

### 8.3 Reproduction Agent

- 코드 저장소 연결
- 환경/의존성 분석
- 재현 체크리스트 생성

### 8.4 Experiment Agent

- baseline 및 ablation 실험안 관리
- 실험 큐 운영
- 추가 실험 제안

### 8.5 Analyst Agent

- 논문 결과와 재현 결과의 차이 분석
- 실패 원인 및 편차 해석
- 인사이트 후보 생성

### 8.6 Proposal / Writing Assistant

기존 문서의 Proposal Agent 개념은 현재 프로토타입에서는 별도 독립 패널보다 다음 기능으로 흡수되어 표현된다.

- 후속 연구 가설 생성
- 연구 제안서 초안 생성
- Writing 단계 초안 작성 보조

즉, Proposal 기능은 살아 있지만 현재 UI에서는 독립 agent보다 결과물 생성 액션으로 노출된다.

## 10. 대표 사용자 플로우

### 시나리오 1. 주제 탐색에서 논문 선택까지

사용자는 연구 질문을 입력한다. Search 화면에서 필터와 relevance 점수를 기반으로 논문을 추린다. preview를 확인한 뒤 논문을 저장하거나 Reading으로 이동한다.

### 시나리오 2. 논문 리딩에서 재현 준비까지

사용자는 Reading 화면에서 핵심 문장을 확인하고 Reader agent가 추출한 repro params를 검토한다. 이후 `Send to Research`로 재현 단계로 이동한다.

### 시나리오 3. 재현과 추가 실험 운영

Research 화면에서 체크리스트를 따라 환경을 준비하고 baseline을 실행한다. 이어서 ablation suggestion 또는 수동 생성으로 추가 실험을 등록한다.

### 시나리오 4. 결과 해석에서 후속 가설 도출까지

Result 화면에서 논문 수치와 재현 수치를 비교한다. Analyst 리포트를 바탕으로 Insight 화면에서 핵심 인사이트와 후속 연구 가설을 정리한다.

### 시나리오 5. 인사이트에서 문서 초안 작성까지

사용자는 Writing 화면에서 연구 결과를 초안 형태로 정리하고 AI 제안을 반영해 문서 완성도를 높인다.

## 11. MVP 포함 범위

프로토타입 기준 현재 MVP에 포함되는 기능은 다음과 같다.

- 프로젝트 기반 연구 워크스페이스
- 논문 검색 결과 탐색과 필터링
- 논문 저장 및 리딩 진입
- 구조화된 논문 리딩
- 재현 파라미터 추출
- 재현 체크리스트
- 실험 목록과 실험 상태 관리
- 결과 비교 리포트
- 인사이트 정리 및 연구 메모
- 후속 연구 가설 생성
- 글쓰기 초안 및 AI 보조

## 12. MVP 제외 범위

프로토타입 기준 아직 명시되지 않았거나 후순위로 보는 항목은 다음과 같다.

## 13. 기존 기획 대비 정렬 변경 사항

이 문서는 다음 관점에서 기존 기획을 조정한다.

### 13.1 5개 레이어에서 6개 워크플로우 단계로 재정의

기존 문서는 수집, 리딩, 재현, 실험 확장, 후속연구의 5개 레이어로 설명했지만, 프로토타입은 `Result`와 `Writing`을 독립된 제품 단계로 명확히 분리한다.

### 13.2 후속연구 단계가 더 구체화됨

기존 문서의 후속연구 레이어는 넓은 개념이었지만, 프로토타입에서는 다음 세 단계로 분해되어 있다.

- `Result`: 실험 비교
- `Insight`: 해석과 가설화
- `Writing`: 문서 초안화

### 13.3 Proposal Agent의 노출 방식 변경

기존 문서는 Proposal Agent를 독립 역할로 강하게 가정했지만, 현재 프로토타입에서는 별도 화면보다 `후속 연구 가설`, `연구 제안서 초안 생성`, `Writing` 보조 기능으로 흡수되어 있다.

### 13.4 협업보다 개인 연구 워크플로우가 우선

기존 문서는 랩 단위 협업을 강조했지만, 현재 프로토타입은 우선 개인 혹은 소규모 팀이 사용하는 연구 워크스페이스에 더 가깝다.

### 13.5 URL 입력 중심 제품이 아니라 워크플로우 중심 제품

기존 MVP 항목은 `논문 URL/arXiv ID 입력`을 강조했지만, 프로토타입은 입력 방식보다 "연구 주제 -> 논문 선택 -> 리딩 -> 재현 -> 결과 -> 인사이트 -> 작성"이라는 상태 기반 워크플로우를 더 중요하게 다룬다.

## 14. 성공 조건

프로토타입 기준 초기 제품이 검증해야 할 핵심 가치는 다음과 같다.

- 논문을 읽고 끝나는 것이 아니라 다음 단계 행동으로 자연스럽게 이동되는가
- 재현 준비와 실험 진행 상태가 명확하게 보이는가
- 결과 차이와 실패가 인사이트로 축적되는가
- 후속 연구 가설과 글쓰기 초안까지 연결되는가

## 15. 한 줄 정의

ARES는 논문 탐색부터 재현, 결과 비교, 인사이트 정리, 글쓰기까지 이어지는 에이전트 기반 연구 실행 워크스페이스다.
