# Search 탭 수집 대시보드 설계안

- 작성일: 2026-04-23
- 대상: ARES `Search` 탭 시작 페이지
- 상태: 사용자 승인 반영 완료

## 1. 설계 목적

Search 탭의 핵심 기능은 프로젝트를 위한 논문을 `검색하고 수집하는 것`이다.  
따라서 Search 탭의 시작 페이지는 빈 검색 결과 화면이 아니라, 현재 프로젝트의 `논문 수집 현황`, `분류 상태`, `수집 논문 목록`을 바로 보여주는 작업 화면이어야 한다.

이 설계안은 Search 탭을 다음 성격으로 정의한다.

- 상단 검색창은 그대로 유지한다.
- 검색창 아래 본문은 `수집 현황 대시보드 + 수집 논문 작업 목록`으로 구성한다.
- 사용자는 Search 탭에서 논문을 검색할 뿐 아니라, 이미 수집한 논문을 `폴더 지정`, `태그 정리`, `PDF 등록`까지 이어서 수행할 수 있어야 한다.

## 2. 범위

### 포함

- Search 탭 시작 페이지 정보 구조 재정의
- 수집 현황 요약 지표
- 차트 세트 정의
- 수집 논문 목록 컬럼/행 액션 정의
- 폴더/태그/PDF 관련 상호작용 정의
- 우측 상세 패널의 보조 역할 정의
- empty state와 반응형 규칙 정의

### 제외

- Agent Search 인터페이스 개편
- Scout 결과 화면 구조 개편
- Reading/Research/Result 단계 화면 개편
- 자동 PDF 수집
- Search 단계에서의 자동 태그 생성

## 3. 핵심 원칙

1. Search 탭 상단 검색창은 유지한다.
2. Search 탭 시작 페이지의 본체는 `검색 결과 미리보기`가 아니라 `수집 현황 대시보드`다.
3. 하단 목록은 보기용 표가 아니라 `수집 논문 작업 테이블`이다.
4. 핵심 정리 작업은 목록에서 바로 끝나야 한다.
5. 우측 상세 패널은 유지하되, 보조 정보층으로 둔다.
6. 태그는 `자동 태그`와 `수동 태그`가 공존한다.
7. PDF는 시스템 자동 수집 대상이 아니라 사용자가 직접 등록하는 자산이다.

## 4. 사용자 가치

이 화면은 사용자에게 아래 가치를 제공해야 한다.

- 지금 프로젝트에서 논문이 얼마나 수집되었는지 즉시 파악할 수 있다.
- 수집된 논문이 폴더/태그/워크플로우 상태 기준으로 얼마나 정리되었는지 읽을 수 있다.
- 정리되지 않은 논문을 목록에서 바로 정리할 수 있다.
- Reading 단계로 보내기 전에 PDF 확보 상태까지 한 번에 확인할 수 있다.

즉 Search 탭은 단순 검색 화면이 아니라 `논문 수집 운영 화면`이 된다.

## 5. 화면 구조

Search 탭 시작 페이지는 아래 3층 구조로 구성한다.

### 5.1 상단 고정 영역

- 현재 검색 입력 UI 유지
- Search 탭의 원래 목적이 여전히 검색이라는 사실을 보존

### 5.2 상단 분석 영역

- 수집 현황 요약 지표
- 분류/분포/워크플로우 상태를 보여주는 차트 세트

### 5.3 하단 작업 영역

- 큰 수집 논문 목록
- 목록 안에서 직접 폴더/태그/PDF 작업 수행
- 행 클릭 시 우측 상세 패널 표시

## 6. 수집 현황 요약 지표

상단 분석 영역의 요약 지표는 아래를 기본으로 한다.

- 총 수집 논문 수
- 폴더가 지정된 논문 수
- Reading queue에 들어간 논문 수
- 태그가 생성된 논문 수
- PDF 수집 완료 논문 수

지표 목적은 “현재 수집 자산이 얼마나 쌓였고 얼마나 정리됐는가”를 한눈에 보여주는 것이다.

## 7. 차트 세트

차트는 `현재 데이터 중심 + 워크플로우 상태 중심` 혼합형으로 설계한다.  
각 차트는 서로 다른 질문에 답해야 하며, 장식용 차트는 두지 않는다.

### 7.1 수집 추이

- 최근 기간별 수집 논문 수 변화
- 질문: 최근 논문 수집이 실제로 진행되고 있는가

### 7.2 폴더별 분포

- 폴더별 논문 수
- 질문: 프로젝트 내 분류가 균형적인가, 특정 폴더에만 몰려 있는가

### 7.3 출처 분포

- venue 기준 또는 year 기준 분포
- 질문: 최근 논문 중심으로 모으고 있는가, 특정 소스에 편향되어 있는가

### 7.4 워크플로우 상태 차트

- 수집됨
- 폴더 미분류
- reading queue
- 태그 생성됨
- PDF 수집됨

질문: 수집한 논문이 다음 단계로 얼마나 흘러가고 있는가

### 7.5 태그 분포

- 자동 태그 + 수동 태그 기준 상위 분포
- 질문: 프로젝트 안에서 현재 어떤 주제 축으로 논문이 쌓이고 있는가

## 8. 하단 수집 논문 목록

하단 목록은 `수집 논문 작업 테이블`로 정의한다.

### 8.1 기본 컬럼

- 논문 제목
- venue / year
- 저장 시점 또는 최근 수집 시점
- 폴더
- 태그 상태
- reading queue 상태
- PDF 상태

### 8.2 PDF 상태 모델

PDF 상태는 2단계로 고정한다.

- 없음
- 수집됨

PDF는 ARES가 자동 수집하는 자산이 아니라, 사용자가 직접 확보하여 붙이는 읽기 준비 자산으로 정의한다.

## 9. 목록에서 가능한 직접 작업

각 행에서 아래 작업을 직접 수행할 수 있어야 한다.

### 9.1 폴더 지정

- 목록 행 안에서 바로 지정/변경
- 드롭다운 또는 셀 편집 패턴 사용

### 9.2 태그 선택

- 목록 행 안에서 바로 수행
- 수동 태그 추가/선택/제거 가능
- 자동 태그는 표시와 필터링 대상이지만 Search 탭에서 직접 수정하지 않음

### 9.3 PDF 등록

- 목록 요소에서 바로 수행
- 드래그앤드롭으로 간편하게 등록 가능
- 드롭 직후 PDF 상태가 `수집됨`으로 갱신

### 9.4 Reading queue 전송

- 수집 논문을 Reading 단계 작업으로 넘기는 액션 유지

### 9.5 상세 패널 열기

- 행 클릭 시 우측 상세 패널 오픈
- 하지만 핵심 정리 작업은 목록 안에서 끝나야 함

## 10. 태그 모델

태그는 `자동 태그`와 `수동 태그`가 공존한다.

### 10.1 자동 태그

- Reading 단계의 AI 요약 시 생성
- 논문의 주제/방법/벤치마크 등 내용적 속성을 반영
- Search 탭에서는 읽기 전용

### 10.2 수동 태그

- 사용자가 Search 탭 목록에서 직접 추가/제거
- 프로젝트 운영 목적의 태그
- 예: `must-read`, `baseline`, `compare-later`, `for-proposal`

### 10.3 표시 원칙

- 목록의 태그 컬럼에는 자동/수동 태그를 함께 보여준다.
- 자동 태그와 수동 태그는 시각적으로 구분한다.
- Search 탭의 inline 작업은 수동 태그에 대해서만 허용한다.

## 11. 우측 상세 패널

우측 상세 패널은 유지하되, 보조 정보층으로 정의한다.

### 역할

- 논문 메타데이터 전체 확인
- abstract 확인
- 태그 전체 확인
- 폴더/queue/PDF 상태 확인
- Save / Read 등 보조 액션 노출

### 원칙

- 주 작업면은 목록이다.
- 패널은 세부 정보 확인과 부가 맥락 제공이 목적이다.
- 목록에서 처리 가능한 작업을 패널 전용으로 옮기지 않는다.

## 12. 필터와 빠른 보기 기준

하단 목록 상단에는 빠른 보기 기준이 필요하다.

- 전체
- 폴더 미분류
- reading 대기
- 태그 생성됨
- PDF 없음
- PDF 수집됨
- 최근 수집

목적은 사용자가 현재 정리해야 할 논문 집합을 빠르게 좁히는 것이다.

## 13. Empty State 규칙

empty state는 단순히 “데이터 없음”을 말하지 않고 다음 행동을 제시해야 한다.

### 수집 논문이 없을 때

- “아직 수집된 논문이 없습니다. 상단 검색으로 첫 논문을 수집하세요.”

### 폴더가 없을 때

- “폴더를 만들고 논문을 분류하면 여기서 분포를 볼 수 있습니다.”

### 태그가 없을 때

- “Reading 이후 자동 태그가 생성되거나, 목록에서 수동 태그를 붙이면 분포가 채워집니다.”

### PDF가 없을 때

- “목록에서 드래그앤드롭으로 PDF를 등록하면 준비도 상태가 갱신됩니다.”

## 14. 반응형 규칙

### 넓은 데스크톱

- 차트는 다열 구성 유지
- 우측 상세 패널 기본 표시 가능

### 중간 폭

- 상세 패널은 접힌 상태를 기본으로 둘 수 있음
- 차트는 2열 유지하되 밀도 축소

### 태블릿

- 상세 패널은 우측 drawer
- 목록 inline 작업은 유지

### 모바일

- 상세 패널은 modal 또는 full-screen sheet
- 차트는 세로 스택
- 목록 inline 작업은 유지하되 터치 친화적으로 단순화 가능

핵심 반응형 원칙은 다음과 같다.

- 주 작업은 언제나 목록에서 끝난다.
- 상세 패널은 보조 정보층이다.
- 차트는 현황을 읽게 하고, 목록은 행동하게 해야 한다.

## 15. 데이터/상태 요구사항

이 설계가 성립하려면 수집 논문 단위로 아래 정보가 존재해야 한다.

- 프로젝트 소속
- 논문 메타데이터
- 저장/수집 시점
- 폴더 지정 상태
- 자동 태그 목록
- 수동 태그 목록
- reading queue 상태
- PDF 수집 여부

Search 탭 시작 페이지의 데이터 기준은 `프로젝트에 수집된 논문 집합`이다.  
즉 시작 페이지는 실시간 검색 결과 집합이 아니라, 프로젝트 라이브러리/수집 자산 집합을 기준으로 렌더링한다.

## 16. 성공 조건

이 화면은 다음 조건을 만족해야 한다.

- 사용자가 Search 탭에 들어오면 현재 수집 현황을 즉시 이해할 수 있다.
- 수집 논문이 어떻게 분류되고 있는지 차트로 읽을 수 있다.
- 정리되지 않은 논문을 목록에서 바로 처리할 수 있다.
- PDF 준비 상태까지 확인하여 Reading 이전 준비도를 판단할 수 있다.
- 우측 상세 패널이 없어도 핵심 수집 정리 작업이 막히지 않는다.

## 17. 최종 요약

이 설계에서 Search 탭 시작 페이지는 더 이상 빈 결과 영역이 아니다.  
대신 `프로젝트의 논문 수집 현황을 보여주고, 수집된 논문을 바로 정리하는 운영 화면`이 된다.

Search 탭의 한 문장 정의는 아래와 같다.

> Search 탭은 프로젝트를 위한 논문을 검색하고, 수집한 논문 자산을 폴더/태그/PDF 기준으로 정리하는 수집 워크스페이스다.
